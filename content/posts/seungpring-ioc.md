개인 프로젝트에서 작은 Spring 스타일 프레임워크를 만들면서 가장 먼저 부딪힌 부분은 객체 생성 책임을 어디에 둘 것인가였다.

처음에는 리플렉션으로 클래스를 찾아 인스턴스를 만들고 필드에 값을 주입하는 정도면 충분하다고 생각했다. 하지만 빈 등록, 의존성 탐색, 순환 참조 감지, 초기화 순서를 한 곳에 몰아넣자 코드가 빠르게 복잡해졌다.

## 컨테이너의 최소 책임

IoC 컨테이너는 모든 것을 아는 거대한 객체가 아니라, 다음 책임을 조율하는 얇은 중심으로 두는 편이 좋았다.

- 빈 정의를 읽는다.
- 필요한 의존성을 계산한다.
- 객체를 생성한다.
- 생명주기 콜백을 호출한다.

이 네 가지가 섞이지 않도록 나누면 테스트하기도 쉬워지고, 나중에 스코프나 프록시 같은 기능을 붙일 자리도 자연스럽게 생긴다.

## 빈 정의와 인스턴스 분리

빈 정의는 설계도이고 인스턴스는 결과물이다. 이 둘을 분리하면 컨테이너가 아직 객체를 만들지 않은 상태에서도 의존성 그래프를 검증할 수 있다.

```java
public record BeanDefinition(
    String name,
    Class<?> beanClass,
    List<DependencyDescriptor> dependencies
) {
}
```

이 구조를 쓰면 생성자 주입과 필드 주입을 같은 방식으로 설명할 수 있고, 실제 객체 생성 전략을 나중에 바꿔도 상위 흐름은 크게 흔들리지 않는다.
